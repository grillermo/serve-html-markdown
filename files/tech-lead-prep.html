Topic 1 — Databases & Redis for Rails Teams

1.1 PostgreSQL vs MySQL: architecture and operational trade-offs

Why interviewers ask this

For a tech lead role, the interviewer is rarely testing whether you have memorized configuration flags. They want to see that you understand why the two engines behave differently under load, that you can predict failure modes before they happen, and that you can guide a team's decisions with reasoning rather than folklore. Every answer in this section should end with an operational consequence — that is what separates a senior engineer's answer from a lead's answer.

The core architectural differences

PostgreSQLMySQL (InnoDB)
MVCC model Old row versions are kept inside the table as dead tuples. VACUUM must run to reclaim the space. Table and index bloat is the classic operational pain point. Old versions live in the undo log and are purged automatically. There is no vacuum, but long-running transactions grow the undo history and slow everything down.
Connection model One OS process per connection, each costing several MB of memory. At scale this practically requires an external pooler such as PgBouncer. One thread per connection — far cheaper. MySQL tolerates thousands of direct connections that would sink a Postgres server.
Storage layout Heap tables. Every index — including the primary key — points to a physical row location, so any secondary index lookup is a single hop. The table is the primary key B-tree (a clustered index). Secondary indexes store the primary key, so every secondary lookup is two B-tree traversals, and a fat primary key bloats every index.
Replication default Physical WAL streaming (byte-level, whole cluster, same major version). Logical replication is available for selective or cross-version needs. Logical binlog replication (row or statement based). GTIDs make failover and topology changes much simpler than they used to be.
Feature depth Transactional DDL, partial and expression indexes, JSONB with GIN indexes, rich types, extensions (PostGIS, pg_stat_statements, pgvector). Simpler operational story, extremely fast primary-key lookups, a huge pool of operators who understand its replication.

Consequences worth stating out loud

  • Random UUIDs as primary keys hurt MySQL badly. Because InnoDB clusters rows by primary key, random inserts land in random pages, causing page splits and cache churn. If you need UUIDs on MySQL, use time-ordered ones (UUIDv7). On Postgres the damage is smaller but index locality still suffers.
  • Transactional DDL is a real developer-experience difference. A failed Postgres migration rolls back cleanly; a failed MySQL migration can leave the schema half-changed, which matters when you are coaching a team on safe deploys.
  • Long transactions are poison in both engines, for different reasons. In Postgres they block vacuum from reclaiming dead tuples (bloat grows); in MySQL they pin the undo log (purge lag grows). Either way, a lead should enforce short transactions and set alerts on transaction age.
  • Postgres connection cost shapes your whole architecture. Puma threads × web dynos + Sidekiq concurrency × worker dynos must fit within max_connections. This single arithmetic problem is behind a large fraction of production incidents in Rails shops.

Q: “Would you pick PostgreSQL or MySQL for a new Rails product?”

A: “Both are excellent and Rails treats them almost identically, so the decision is organizational as much as technical. I default to PostgreSQL for its feature depth — JSONB, partial indexes, transactional DDL, and the extension ecosystem — which pays off as the product grows. I would choose MySQL when the team already has deep MySQL operational expertise or the platform team standardizes on it, because operational familiarity prevents more incidents than any feature difference. What matters more than the choice is that the team understands its engine's failure modes: vacuum and connection limits on Postgres, clustered-index and replication-topology implications on MySQL.”

Framing habit for the whole interview: answer database questions in the shape mechanism → consequence → what I would do about it as a lead. For example: “Postgres keeps dead tuples in the heap (mechanism), so unindexed churn causes bloat and slow scans (consequence), so I make sure autovacuum is tuned per hot table and we alert on dead-tuple ratios (leadership action).”

1.2 Indexing & Query Performance

Index fundamentals you should be able to teach

  • B-tree is the default and handles equality, ranges, and sorting. Composite indexes obey the leftmost-prefix rule: an index on (account_id, created_at) serves WHERE account_id = ? and WHERE account_id = ? AND created_at > ?, but not a query filtering on created_at alone.
  • Column order: equality-filtered columns first, then the range or sort column. “Most selective first” is a common half-truth — the access pattern of your real queries decides the order.
  • Partial indexes (Postgres): CREATE INDEX ... WHERE deleted_at IS NULL. They are small, fast to scan, and cheap to maintain — ideal for soft deletes and state-machine columns like WHERE status = 'pending' where only a sliver of rows ever match.
  • Expression indexes: if a query wraps a column in a function — LOWER(email) — a plain index on email is unusable. Index the expression itself, or better, normalize on write (or use citext).
  • Covering indexes / index-only scans: Postgres INCLUDE (col), or in MySQL putting every selected column in the index, lets the engine answer from the index without touching the table. In Postgres this depends on the visibility map, which is another reason vacuum health matters.
  • GIN indexes (Postgres) serve JSONB containment queries, array membership, and full-text search. Mention them when the conversation turns to “we store flexible attributes.”
  • Indexes are not free: each one slows every write and consumes RAM that competes with your working set. Audit for unused indexes with pg_stat_user_indexes and drop them deliberately.

Reading a query plan under pressure

EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
  • Seq Scan on a large table under a selective filter usually means a missing or unusable index. Check for functions wrapping the column, type mismatches (string compared to integer), or an OR that defeats the planner.
  • Estimated rows wildly different from actual rows means stale statistics. Run ANALYZE, and for skewed columns raise the statistics target. Bad estimates cascade into bad join choices.
  • Nested Loop joins over large row counts often follow from those bad estimates; a Hash Join is usually what you want for bulk work.
  • BUFFERS output tells you whether the query is hitting shared memory or disk — a query can be “fast” in a warm cache and terrible after a restart or failover.
  • Fleet-wide, do not guess: rank real production queries by total time with pg_stat_statements (Postgres) or the slow query log and performance_schema (MySQL), then fix the top of the list.

Rails-specific performance gotchas

  • N+1 queries — the eternal one. Fix with includes / preload / eager_load; prevent regressions with strict_loading on hot models and query-count assertions in tests. Know the difference: preload always issues separate queries, eager_load always joins, includes lets Rails decide.
  • COUNT(*) on large tables is a full-index scan every time. Use counter caches for exact per-parent counts, or an estimate from pg_class.reltuples when “about how many” is acceptable.
  • Deep OFFSET pagination reads and discards every skipped row; page 5,000 is a table scan. Switch to keyset (cursor) pagination ordered by an indexed, unique column.
  • first_or_create races under concurrency produce duplicates. The database is the only reliable referee: unique index plus rescue-and-retry, or an upsert/insert ... on conflict.
  • Ordering without an index — an ORDER BY created_at on a filtered set can force a sort node; a composite index ending in the sort column removes it.

Q: “A page went from 50 ms to 3 seconds after a deploy. Walk me through your process.”

A: “First reproduce it and confirm scope — one page or sitewide, one tenant or all. Then I go to the APM trace or slow-query log to find the offending SQL rather than reading the diff line by line. I run EXPLAIN ANALYZE on that query and compare the plan against expectations: is the index still being used, are the row estimates sane? The usual culprits are a new N+1 from a changed association, a query that lost its index because a new filter column or function was introduced, or lock contention from a long transaction or migration that shipped in the same deploy. Once fixed, I add a guard so it cannot silently return — a query-count assertion in tests and a dashboard on that endpoint's p95.”

One-liner that lands well: “I treat pg_stat_statements as the to-do list and EXPLAIN ANALYZE as the debugger. Optimizing queries nobody runs is a hobby, not engineering.”

1.3 Replication

Replication modes and their trade-offs

  • Asynchronous (the default everywhere): the primary commits without waiting for replicas. Writes stay fast, but replicas lag behind, which means stale reads — and if the primary dies before a replica catches up, the un-replicated transactions are lost on failover.
  • Synchronous: the commit waits for at least one replica to acknowledge. You trade write latency for a zero-data-loss guarantee. In Postgres this is synchronous_standby_names; MySQL offers semi-synchronous replication. Most teams run one synchronous standby at most, because every additional one compounds latency and availability risk.
  • Physical vs logical (Postgres): physical replication streams the WAL byte-for-byte — the whole cluster, same major version, replicas are exact copies. Logical replication publishes row changes per table — it can cross major versions and replicate selectively, which is why it powers near-zero-downtime major upgrades.
  • MySQL specifics worth naming: row-based binlog format is the safe default (statement-based can drift on non-deterministic statements); GTIDs give every transaction a global ID so replicas can be repointed without binlog-position archaeology.

Read replicas in Rails — know this cold

# config/database.yml defines primary and replica entries
class ApplicationRecord < ActiveRecord::Base
  connects_to database: { writing: :primary, reading: :replica }
end

# Explicit routing for heavy read work:
ActiveRecord::Base.connected_to(role: :reading) do
  Report.heavy_aggregation
end
  • Automatic role switching (Rails 6+): GET and HEAD requests read from the replica; anything that writes goes to the primary. After a write, the middleware pins that user's session to the primary for a configurable delay (2 seconds by default) — the built-in mitigation for read-your-own-writes.
  • Replication lag is the number one gotcha. A user creates a record, the next request reads the replica, and the record is “missing.” Beyond session pinning, mitigations include reading the primary for the specific model involved in the flow, monitoring lag and disabling replica reads past a threshold, and designing flows so the response carries the data the next page needs.
  • What causes lag spikes: large writes and backfills, vacuum activity on the primary, long-running queries on the replica that conflict with WAL replay, and under-provisioned replica I/O. Batch backfills with sleeps between batches, and know that hot_standby_feedback trades replica query cancellation for primary-side bloat.
  • Replicas scale reads only. Every replica still replays every write. If the write path is the bottleneck, replicas do nothing for you — that is the segue into sharding.

Failover and the lead's responsibilities

  • Managed platforms (RDS, Aurora, Cloud SQL) or Patroni handle promotion mechanics, but a failover that has never been rehearsed is a hypothesis, not a capability. Schedule game days; measure the real RTO.
  • Know your RPO (how much data you can lose — decided by sync vs async) and RTO (how long promotion plus DNS/proxy cutover takes) and make sure the business has agreed to those numbers.
  • After failover, the application layer matters: connection pools must reconnect cleanly, in-flight transactions will fail and must be retried or surfaced honestly, and Sidekiq jobs should be idempotent precisely because they will be re-run around such events.
  • Backups are a separate discipline from replication: replication happily replicates a DELETE of your users table. Point-in-time recovery via WAL archiving is the safety net; test restores, not backups.

Q: “When do you add a read replica versus a cache versus a bigger box?”

A: “In order of cheapness. First fix queries and indexes — the biggest wins are almost always there. Second, cache what is genuinely cacheable, because a cache hit is cheaper than any database read. Third, scale vertically; it is boring and buys time with zero added complexity. Fourth, add read replicas when read volume truly saturates the primary and the product can tolerate staleness. Sharding comes last, only when write volume or data size exceeds a single primary. Each step adds a failure mode — lag, invalidation, topology — so I do not let the team skip ahead for excitement.”

1.4 Partitioning & Sharding

Before you shard: the escalation ladder

  1. Partitioning (Postgres declarative partitioning, typically by time or tenant): one logical table split into physical children inside the same database. Queries that hit the partition key prune to one child; old partitions can be dropped instantly instead of running a months-long DELETE. This solves “the table is huge” without any of sharding's distributed-systems costs.
  2. Archiving: move cold rows to cheap storage or a warehouse. Many “we need to shard” conversations end when someone notices 90% of the table is never queried.
  3. Functional decomposition: move a hot, self-contained domain (analytics events, audit logs, notifications) into its own database. Rails multi-database support makes this nearly free, and it isolates the noisy workload's I/O and vacuum behavior from the core schema.
  4. Horizontal sharding — only when a single primary cannot absorb the write volume or hold the data set even after all of the above.

Sharding strategies

StrategyHow it worksStrengths / weaknesses
Tenant / hash Shard key is something like account_id; a hash or explicit assignment maps each tenant to a shard. Natural fit for B2B SaaS — each tenant's data is colocated, so most queries stay single-shard. Weakness: one whale tenant can outgrow its shard, so you need a plan for moving tenants.
Range Split by ranges of the key, commonly time. Trivial to add new shards for new ranges. Weakness: the newest range is where all the writes go — a built-in hot spot.
Directory / lookup A mapping service or table records which shard owns each key. Maximum flexibility — tenants can be moved by updating the map. Weakness: an extra hop on every request, and the directory becomes critical infrastructure with its own availability story.

Rails-native horizontal sharding

class ApplicationRecord < ActiveRecord::Base
  connects_to shards: {
    default: { writing: :primary, reading: :primary_replica },
    shard_two: { writing: :shard_two, reading: :shard_two_replica }
  }
end

ActiveRecord::Base.connected_to(shard: :shard_two) do
  Account.find(id)   # runs against shard_two
end

Rails 6.1+ supports this natively, and Rails 7's ShardSelector middleware can resolve the shard per request (from subdomain, session, or token). The framework routes connections; choosing the shard key and living with its consequences is still entirely your problem.

The costs — volunteer these before being asked

  • No cross-shard joins or transactions. Anything that spans shards becomes application-level orchestration, with all the partial-failure handling that implies.
  • Cross-shard reporting needs scatter-gather queries or, more realistically, an ETL pipeline into a warehouse. Budget for this on day one.
  • Global uniqueness: auto-increment IDs collide across shards. Use UUIDv7 or Snowflake-style IDs — time-ordered, so they stay index-friendly.
  • Choosing the wrong shard key is a company-level mistake. Resharding live data is a migration project measured in quarters. Pick the key that matches how the product actually queries data — almost always the tenant.
  • Every migration now runs N times, and observability must be per-shard: one hot shard hides comfortably inside healthy averages.
  • Middleware alternatives: Citus (Postgres extension), Vitess (MySQL), and PlanetScale push shard routing below the application. App-level sharding keeps you in control but smears complexity across every query path — a genuine trade-off to discuss, not a default.

Q: “Your primary is at 90% write capacity and growing 10% a month. What is your plan?”

A: “That growth rate gives me roughly a year of runway, so the plan has parallel tracks. Immediately: verify the writes are real — eliminate redundant updates, batch counters, move ephemeral write-heavy data like sessions or view counts out of Postgres entirely. Short term: vertical headroom and partitioning of the largest tables. In parallel, because sharding takes quarters, I would start the design now: pick the shard key (almost certainly tenant), introduce it into every table and query path, build the routing layer behind a flag, and migrate tenants gradually — dual-writing during the transition. The worst outcome is starting that project with three weeks of runway left.”

1.5 Database Bottleneck Playbook

Symptom → cause → fix

SymptomLikely causeFix
High CPU on the database Missing indexes forcing sequential scans; N+1 storms multiplying cheap queries into expensive aggregate load Rank by total time in pg_stat_statements; index or rewrite the top offenders; fix N+1 in the app layer
Connection errors, checkout timeouts Pool exhaustion: (Puma threads × web processes) + (Sidekiq concurrency × workers) exceeds max_connections Do the arithmetic explicitly; right-size the Rails pool; put PgBouncer in transaction mode in front of Postgres
Writes stalling, lock wait timeouts Lock contention: long transactions, DDL without CONCURRENTLY, hot rows such as shared counters Inspect pg_locks joined to pg_stat_activity; shorten transactions; set lock_timeout in migrations; move hot counters to Redis or batch them
Postgres gradually slower; tables far larger on disk than their data Bloat — autovacuum falling behind churn, often abetted by long idle-in-transaction sessions Tune autovacuum per hot table (lower scale factors); pg_repack to rebuild without long locks; alert on idle-in-transaction age
Replica lag spikes Bulk writes or backfills, vacuum on the primary, long replica-side queries conflicting with WAL replay Batch backfills with pauses; throttle jobs on measured lag; weigh hot_standby_feedback trade-offs
Periodic latency spikes at regular intervals Checkpoints, autovacuum runs, or scheduled jobs Correlate spike timing with server activity; spread checkpoints out; move heavy cron work off-peak
One endpoint slow only for the biggest customers Data skew: plans and indexes that work at median cardinality collapse at the 99th percentile tenant Test against production-scale data for the largest tenant; add composite indexes for their access pattern; consider partitioning by tenant

Migrations at scale — a signature tech-lead topic

  • Adding an index: plain CREATE INDEX blocks writes for the duration. Always algorithm: :concurrently in Postgres, which requires disable_ddl_transaction! in the Rails migration. MySQL 8 does most index builds online.
  • Adding a column: safe and instant in modern versions — including with a constant default (Postgres 11+ stores the default in metadata). Adding a column with a volatile default or backfilling in the same migration rewrites the table under lock: never do it.
  • Backfills belong in batched background jobs (find_in_batches, sleep between batches, resumable), never inline in the migration that deploys the schema.
  • NOT NULL / constraints on big tables: add the constraint as NOT VALID, backfill, then VALIDATE CONSTRAINT — validation takes a much weaker lock.
  • Set lock_timeout so a migration that cannot get its lock fails fast instead of queueing behind a long transaction while every other query queues behind it — that pile-up is the classic “the migration took the site down” incident.
  • Institutionalize it: the strong_migrations gem turns all of the above from tribal knowledge into an automatic lint. Recommending it is a leadership answer, not a tooling trivia answer.

Prevention — what a lead puts in place

  • Dashboards and alerts on: p95 query time, connection count vs limit, replica lag, oldest transaction age, dead-tuple ratio, queue latency.
  • CI guards: query-count assertions on hot endpoints, strict_loading on core models, strong_migrations.
  • A written runbook per symptom above — during an incident nobody should be composing pg_stat_activity queries from memory.
  • Load-test with production-shaped data. Almost every “it was fine in staging” story is a cardinality story.
Interview framing: always answer bottleneck questions with the loop measure → hypothesize → fix → verify → prevent, and name real tools at each step (APM, pg_stat_statements, EXPLAIN ANALYZE, pg_locks). The “prevent” step is what distinguishes a lead from a strong senior.

1.6 Redis: Caching, Sidekiq, and Choosing the Right Store

The decision framework — lead with this

Use PostgreSQL/MySQL when…Use Redis when…
The data is the source of truth: you need durability, transactions, constraints, joins, and the ability to query it in ways you haven't thought of yet. The data is derived, ephemeral, or reconstructible, and you need sub-millisecond latency or atomic operations on shared state (counters, queues, locks, leaderboards).
Losing it is an incident: orders, accounts, money, anything you would have to apologize for. Losing it is degraded performance (a cold cache) or bounded rework (re-enqueued jobs — with caveats below).

Sound-bite: “Redis is memory-speed shared state; the relational database is durable truth. If I can't cheaply rebuild the data, it doesn't belong only in Redis.” Redis persistence (RDB snapshots, AOF) exists, but it is a safety net — not a durability guarantee on par with a WAL-based RDBMS, and managed cache offerings often run with eviction enabled.

Caching in Rails

  • Cache in layers, highest first: HTTP caching (CDN, stale?, ETags) → fragment caching → Rails.cache object caching → per-request memoization. A CDN hit costs you nothing; a memoized value still cost a request.
  • Key-based expiration is the Rails idiom: cache keys embed updated_at (cache [post, current_user]), so you never explicitly invalidate — a change produces a new key and old entries age out via LRU. Russian-doll caching nests fragments, with touch: true on associations cascading key changes upward so an updated comment busts its post's fragment.
  • Explicit invalidation is where the bugs live. Prefer key-based expiration; treat every Rails.cache.delete in a code review as a question to answer.
  • Cache stampede: a hot key expires and hundreds of requests recompute it simultaneously, sometimes hard enough to take the database down. Mitigations: race_condition_ttl, jittered TTLs, lock-and-recompute, serve-stale-while-revalidate.
  • Eviction policy is a configuration decision with teeth: a cache Redis should run allkeys-lru (or LFU); a Sidekiq Redis must run noeviction, because an evicted key there is a lost job. This is why cache and queues get separate Redis instances — a classic interview checkpoint.
  • Worth mentioning: Rails 7.1+ ships Solid Cache and Solid Queue, database-backed alternatives — the “do we even need Redis?” trend. A balanced take: fine for many apps and one less system to operate; Redis still wins on latency and rich data structures.

Sidekiq and Redis as a queue

  • Mechanics: jobs are JSON pushed onto Redis lists; workers block-pop them. The scheduled and retry sets are sorted sets scored by timestamp.
  • Delivery is at-least-once. A worker can complete the work and die before acknowledging, so the job runs again. Therefore jobs must be idempotent — the single most important sentence in this section. Techniques: uniqueness checks on natural keys, upserts, state-machine guards, idempotency keys for external side effects like charges and emails.
  • Reliability tiers: open-source Sidekiq can lose a job that was fetched but unfinished when a process is hard-killed; Sidekiq Pro's super_fetch recovers such orphans. Knowing this trade-off exists is the point.
  • The classic race: enqueue a job inside a transaction; the job runs before the commit and raises RecordNotFound. Fix: enqueue in after_commit, or rely on Rails 7.2's enqueue_after_transaction_commit. The heavyweight relative is the transactional outbox pattern.
  • Queue design: organize queues by latency SLO (critical, default, low), not by domain. Monitor queue latency (age of the oldest job), not queue size — a million-job queue draining fast is healthier than a ten-job queue stuck for an hour. Alert on the dead set.
  • Sizing gotcha: every Sidekiq thread holds a database connection. Sidekiq concurrency must fit the ActiveRecord pool, and total workers × concurrency counts against max_connections — the connection arithmetic from section 1.1 again.

Other legitimate Redis uses — and the traps

  • Good: rate limiting (INCR with EXPIRE, or sliding windows), distributed locks (SET NX PX — mention Redlock's caveats if pressed), leaderboards and counters via sorted sets, ActionCable pub/sub, ephemeral sessions, presence tracking.
  • Traps to call out unprompted: using a cache Redis as a source of truth; sharing one Redis between cache and queues (eviction eats jobs); unbounded keys with no TTL; KEYS * in production (use SCAN); assuming Redis Cluster gives you multi-key transactions across slots.

Q: “Checkout is slow. Would you cache it in Redis?”

A: “Not the money path — correctness beats speed there. First I would profile: is it N+1 queries, an unindexed lookup, a synchronous external call to a payment or tax API? Slow side effects — email, webhooks, analytics — move to Sidekiq. The read-mostly inputs, like the product catalog or tax tables, are safe to cache with key-based expiration. The mutable order and payment state never is. Then verify with p95 before and after.”

Q: “How do you keep the job queue healthy during a traffic spike?”

A: “Alert on queue latency per SLO tier, not size. Autoscale workers within the database connection budget. Shed or delay low-priority queues so critical work drains first. Keep jobs small and idempotent so we can drain aggressively and retry safely. And watch Redis memory — under noeviction, a full Redis means enqueues start failing, so the alert has to fire well before that.”

Closing themes to project throughout: (1) measure before optimizing; (2) the simplest thing that scales — indexes → cache → vertical → replicas → shard, in that order; (3) durability boundaries decide the store, not fashion; (4) name the failure modes unprompted — replica lag, cache stampede, at-least-once delivery, connection exhaustion. Leads get hired for knowing where things break.

Topic 2 — Advanced Ruby & Deep Rails Internals

2.1 ActiveRecord Performance

The N+1 problem, precisely stated

An N+1 occurs when you load a collection of N records with one query, then trigger one additional query per record while iterating — usually by touching an association that was never loaded. The insidious part is that each individual query is fast, so nothing shows up in slow-query logs; the damage is aggregate latency and connection-pool pressure.

# 1 query for posts, then N queries — one per post — for authors
Post.limit(50).each { |post| puts post.author.name }
# SELECT * FROM posts LIMIT 50
# SELECT * FROM users WHERE id = 1
# SELECT * FROM users WHERE id = 2
# ... 48 more

includes vs. preload vs. eager_load — the canonical answer

All three eliminate N+1s by eager-loading associations, but they generate fundamentally different SQL:

MethodSQL strategyCan you reference the association in where/order?When to use
preload Always two (or more) separate queries. One for the parent, one WHERE id IN (...) per association. No — raises StatementInvalid if you try, because the joined table isn't in the query. Default choice. Cheapest for has_many, avoids row multiplication and wide result sets.
eager_load Always a single query with LEFT OUTER JOIN and aliased columns for every table. Yes — the association's table is present in the query. When you must filter or sort the parent by the association's columns and still want it loaded.
includes Delegates. Uses the preload strategy by default, but silently switches to eager_load (LEFT JOIN) if it detects a reference to the association — via references(:assoc) or a string/hash condition mentioning it. Yes, if combined with references. Fine as a general-purpose tool, but its adaptive behavior means the SQL can change under your feet when someone adds a where clause later. 
# preload: two queries, cannot filter on authors
Post.preload(:author).where(published: true)

# eager_load: one LEFT OUTER JOIN, can filter on authors
Post.eager_load(:author).where(users: { admin: true })

# includes: preload by default...
Post.includes(:author)                                # 2 queries
# ...but flips to eager_load when referenced:
Post.includes(:author).where(users: { admin: true }).references(:author)  # 1 JOIN query
The tech-lead-level nuance: eager_load on a has_many multiplies rows (a post with 20 comments returns 20 rows that ActiveRecord must de-duplicate in Ruby), and it selects every column of every joined table, inflating the result set and object allocation. That's why preload is usually faster for one-to-many associations even though it issues more queries. Also mention joins: it produces an INNER JOIN for filtering only — it does not load the association, so Post.joins(:author).each { |p| p.author } is still an N+1. A common bug is "fixing" an N+1 with joins and wondering why Bullet still complains.

Detecting and preventing N+1s systematically

  • Bullet gem in development/test — flags both N+1s and unused eager loading (over-fetching is also a smell). Can be configured to raise in CI so regressions fail the build.
  • Prosopite — a newer alternative that detects N+1s by fingerprinting query patterns rather than instrumenting associations; fewer false negatives, and it can be scoped to request middleware and test suites.
  • Strict loading (Rails 6.1+) — the framework-native guardrail. has_many :comments, strict_loading: true, per-record post.strict_loading!, per-query Post.strict_loading, or globally via config.active_record.strict_loading_by_default = true. Lazy-loading then raises ActiveRecord::StrictLoadingViolationError (or logs, with strict_loading_mode = :log) instead of silently querying. As a lead I'd advocate enabling it by default in new apps and per-association in legacy ones.
  • APM (Datadog, Scout, Skylight, rack-mini-profiler) in production — N+1s that only manifest with production data shapes show up as repeated similar queries within one trace.

pluck vs. select — and their relatives

# select: returns AR model instances with only those attributes populated.
# Stays a Relation — chainable, lazy, usable as a subquery.
User.select(:id, :email).where(active: true)

# pluck: fires SQL immediately, returns raw values (Array), no AR objects built.
User.where(active: true).pluck(:email)   # => ["a@x.com", "b@x.com"]
selectpluck

Related tools worth naming unprompted: pick(:email) (pluck limited to one row), ids (shorthand for pluck(:id)), exists? (SELECT 1 ... LIMIT 1 — never use .count > 0 or .present? which loads records), find_each/in_batches for iterating large tables without loading everything into memory, update_all/insert_all/upsert_all for bulk writes that skip callbacks and validations (say explicitly that skipping callbacks is both the feature and the danger), and counter_cache / size vs count vs length (size uses the counter cache or loaded collection; count always queries; length always loads).

Likely follow-up: "A page is slow. Walk me through diagnosing it." — Answer in layers: reproduce with production-like data; read the request trace in APM or rack-mini-profiler; check query count first (N+1s), then individual query cost (EXPLAIN ANALYZE, missing indexes, sequential scans), then allocation volume (are we instantiating 50k AR objects to render 20 rows?); only then reach for caching. Caching an N+1 is hiding a defect.

2.2 Concurrency & Memory

The GVL (Global VM Lock — the modern name for the GIL)

CRuby allows only one thread to execute Ruby bytecode at a time per process. Threads therefore give you concurrency but not parallelism for CPU-bound Ruby code. The crucial exception: the GVL is released during blocking I/O — database calls, HTTP requests, file reads — and inside many C extensions. Since a typical Rails request spends most of its wall time waiting on the database and external services, threads still deliver real throughput gains for web workloads.

How this shapes Puma configuration

Puma is a hybrid: worker processes (forked, true parallelism, COW memory sharing) each running a thread pool (concurrency during I/O wait). The classic reasoning an interviewer wants to hear:

  • More threads per worker → better utilization while requests wait on I/O, but: higher peak latency for CPU-bound requests (they queue behind the GVL), more DB connections needed (pool size must be ≥ threads per process or you get ConnectionTimeoutError), and more memory-arena fragmentation.
  • Rails 7.2 changed the default from 5 threads to 3 threads per worker, following measurements (largely from Shopify/basecamp discussions, GitHub issue #50450) that high thread counts hurt p95 latency more than they helped throughput for typical apps.
  • Workers ≈ number of cores (adjusted for memory budget). Use preload_app! so the app is loaded before forking, maximizing copy-on-write sharing; pair with before_fork/on_worker_boot to disconnect/reconnect the database, since file descriptors and sockets must not be shared across fork.

Ruby's Garbage Collector — what to actually say

  • Mark-and-sweep, generational (RGenGC, Ruby 2.1+): objects that survive some minor GCs are promoted to the old generation; minor GCs scan only the young generation plus a "remembered set" of old objects that reference young ones (tracked via write barriers). Major GCs scan everything and are the expensive ones.
  • Incremental marking (2.2+): major-GC mark phases are sliced into short steps interleaved with execution to cut pause times.
  • Compaction (GC.compact, 2.7+; auto-compaction option 3.0+): defragments the heap by moving objects, improving copy-on-write friendliness across Puma forks.
  • Structure: the heap is made of fixed-size slots (RVALUE, historically 40 bytes) in heap pages; Variable Width Allocation (3.2+) lets objects occupy multiple slot sizes, reducing off-heap malloc for things like short strings.
  • Observability: GC.stat (watch major_gc_count, heap_live_slots, old_objects), GC::Profiler, ObjectSpace.memsize_of_all, and gems like memory_profiler, derailed_benchmarks (bundle exec derailed bundle:mem to find gem bloat at boot), and stackprof/vernier for CPU.

Memory bloat: the classic production narrative

Rails processes commonly grow from ~300 MB to 1 GB+ over hours and plateau — usually not a leak but heap fragmentation. glibc malloc creates per-thread memory arenas; multi-threaded Puma workers scatter allocations across arenas, and freed memory can't be returned to the OS because pages are partially occupied. The standard remediations, in the order I'd try them:

  1. MALLOC_ARENA_MAX=2 — caps glibc arenas; cheap, often a 10–30% resident-memory win.
  2. jemalloc (LD_PRELOAD or compiling Ruby with --with-jemalloc) — dramatically better fragmentation behavior for multithreaded Ruby; commonly halves memory. This is the well-known "one weird trick."
  3. Distinguish bloat (one request allocating a huge working set — e.g., loading 200k AR objects for a report; fix the query, stream, or batch) from a true leak (monotonic growth without plateau — bisect with memory_profiler, look at C extensions, thread-locals, and ever-growing memoized caches).
  4. Mitigations while root-causing: worker recycling (puma_worker_killer), horizontal capacity headroom. Be explicit that this is a tourniquet, not a fix, and schedule the real investigation.

Bonus points: mention YJIT (production-ready since 3.2, big wins in 3.3/3.4) — pay a modest memory cost per process (--yjit-exec-mem-size) for 15–30% latency improvements on typical Rails workloads.

2.3 Metaprogramming

The mechanics (know them cold, state them briefly)

class Settings
  def initialize(data) = @data = data

  # Lazy interception. MUST pair with respond_to_missing?
  def method_missing(name, *args, &block)
    key = name.to_s
    @data.key?(key) ? @data[key] : super   # always call super for unknowns
  end

  def respond_to_missing?(name, include_private = false)
    @data.key?(name.to_s) || super
  end
end

class Event
  STATES = %w[draft published archived].freeze

  # Eager definition: real methods, visible to respond_to?, tools, and profilers
  STATES.each do |state|
    define_method("#{state}?") { status == state }
  end
end
  • method_missing — invoked as the last stop of method lookup. Failing to call super swallows NoMethodErrors; failing to define respond_to_missing? breaks respond_to?, method(:x), and anything that introspects.
  • define_method — defines a genuine method whose body is a closure; slightly slower to call than a def, but eager, greppable-ish, and doesn't hijack dispatch.
  • class_eval / instance_eval — reopen a class vs. change self for one object; the backbone of DSLs.
  • instance_variable_get/set, send/public_send — prefer public_send when the method name is dynamic to preserve visibility rules.

When to use it — and the maintenance-cost argument

The tech-lead framing: metaprogramming trades write-time convenience for read-time cost, and a team reads code far more often than it writes it. My decision rules:

  • Legitimate uses: framework/library-level DSLs where the pattern repeats hundreds of times and is documented (this is exactly what Rails does — has_many generates ~a dozen methods via modules and class_eval; ActiveModel attributes; delegate), reducing genuinely mechanical duplication (state predicate methods above), test support, adapters over dynamic external schemas.
  • Prefer define_method over method_missing whenever the method set is enumerable up front: real methods appear in respond_to?, methods, stack traces, and profilers. method_missing is for genuinely open-ended interfaces (e.g., OpenStruct-likes, dynamic finders of old).
  • The nightmare scenarios to volunteer: (1) grep-ability — define_method("#{prefix}_#{suffix}") means a developer searching for archived_at_changed? finds nothing; (2) debugging — stack traces point into eval'd strings or generic dispatchers; (3) method_missing on a hot path is slow (full lookup miss every call) and hides typos as silent nil-ish behavior; (4) monkey-patching core classes or gems creates upgrade landmines — if unavoidable, isolate patches in clearly named files (lib/core_ext/), comment the gem version they target, and prefer Module#prepend over alias-method chains so super still works.
  • Team policy: metaprogramming in application code needs a justification in review; in a shared internal gem with docs and tests, the bar is lower. If a new hire can't trace where a method comes from within a minute (method(:foo).source_location should work!), it's too clever.
Likely follow-up: "How does Rails itself use metaprogramming?" — Associations generate methods into an anonymous module included in the model (so you can override and call super); ActiveSupport::Concern manages dependency ordering of included modules; attribute methods are lazily defined on first use via define_attribute_methods; routing helpers are generated methods; delegate compiles a real method with module_eval for speed. Knowing that Rails generates real methods in modules rather than relying on method_missing (it only uses that as a first-call fallback) is exactly the under-the-hood detail this question fishes for.

Topic 3 — The "Code Review" & Technical Alignment

3.1 Design Patterns & Code Smells

How to run the live code-review exercise

They're evaluating how you elevate engineers, not just whether you spot flaws. Structure your critique:

  1. Orient first: "Let me read this end-to-end before commenting." Say what the code is trying to do — proves comprehension before criticism.
  2. Triage by severity: correctness/security bugs → performance issues → design concerns → style. Explicitly say "the style points I'd leave to RuboCop, not humans."
  3. Ask questions instead of decreeing: "What happens if the payment succeeds but this second save fails?" beats "this is wrong."
  4. Praise something genuine. Leads who only find faults create review-averse cultures.

The messy controller — smells to name on sight

class OrdersController < ApplicationController
  def create
    @order = Order.new(params[:order].permit!)                 # 1. permit! = mass assignment hole
    @order.total = @order.items.sum { |i| i.price * i.qty }    # 2. domain logic in controller
    if @order.save
      Payment.charge!(current_user.card_token, @order.total)   # 3. external call after save,
      OrderMailer.confirmation(@order).deliver_now             #    no transaction, deliver_now blocks
      current_user.update(order_count: current_user.order_count + 1) # 4. race condition
      redirect_to @order
    else
      render :new
    end
  rescue Payment::Error
    render :new                                                # 5. order saved but payment failed —
  end                                                          #    inconsistent state, swallowed error
end

Named smells: fat controller, feature envy (controller computing order math), missing transactional boundary, non-atomic increment (update_counters / increment_counter exists), synchronous side effects in the request cycle (mail should be deliver_later; payment ideally goes through an idempotent job or an outbox), permit!, exception swallowed without user feedback or observability.

3.2 Testing Philosophy

Opinionated but pragmatic — a defensible position

  • Shape: roughly a pyramid — lots of fast unit/model tests, a solid layer of request/integration specs (in Rails these are the workhorse: they cover routing, controller, model, and serialization cheaply), and a thin set of system/browser tests for critical money paths only (signup, checkout). System tests are slow and flaky-prone; every one must pay rent.
  • TDD: I use it where it shines — pure logic, bug fixes (write the failing test first; it proves you understood the bug and pins the regression), and API design (the test is the first client). I don't demand test-first for exploratory UI work; I demand tests with the PR. Dogma loses teams; the discipline of "no behavior change without a test" is the actual invariant.
  • RSpec vs. Minitest: both fine; consistency beats preference. RSpec has richer matchers, better failure output, and dominates hiring familiarity; Minitest is faster to boot, plain Ruby, less DSL magic. I'd never migrate an existing suite between them — negative ROI.

Mocks and stubs — the trade-off answer

  • Detroit/classicist school: use real objects, assert on state. Tests survive refactoring; slower, and failures can be non-local.
  • London/mockist school: isolate the unit, assert on interactions. Fast, pinpoint failures; but tests couple to implementation — refactor a collaborator's API and 40 tests break while behavior is unchanged. Worst case: heavily mocked suites that pass while production is broken, because mocks encode your assumptions about collaborators rather than their behavior.
  • My working rules: mock at system boundaries you don't own (HTTP APIs — via WebMock/VCR or better, a hand-written fake client with its own contract tests; payment gateways; the clock via travel_to). Don't mock what you own and can cheaply instantiate; never mock the object under test; avoid stubbing ActiveRecord internals (allow(User).to receive(:find) is a suite that tests nothing). verify_partial_doubles = true so stubs of nonexistent methods fail loudly.
  • Factories: FactoryBot over fixtures for clarity, but the lead-level concern is factory bloat — a default factory dragging five associations means "unit" tests hit the DB 30 times per example. Keep default factories minimal; use traits for elaboration; consider build_stubbed where persistence isn't needed.

Fast CI/CD as a leadership responsibility

CI time is a team-productivity multiplier and it's the lead's job to defend it. Concrete levers, roughly in ROI order:

  1. Parallelize: split the suite across containers (Knapsack Pro, or CI-native splitting balanced by historical timings), and use Rails' native parallelize(workers: :number_of_processors) within a node.
  2. Profile the suite like production code: --profile for slowest examples; the top 10 are usually system tests or factory cascades.
  3. Kill flakes with process: quarantine tag + tracked issue + weekly triage; a retry-until-green culture destroys trust in CI and eventually ships real regressions. Most Rails flakes are time (travel_to), order dependence (run with random seed always, keep the seed in the failure output), leaked state, or Capybara wait misuse (sleep in tests is a code smell — use matchers' built-in waiting).
  4. Structural: cache gems/assets, fail fast on lint before the heavy suite starts, and keep the pipeline under ~10 minutes to merge — beyond that, developers batch changes and quality drops.

3.3 Upgrade & Maintenance Strategies

The playbook for upgrading a legacy Rails app (e.g., 6.1 → 7.x → 8)

  1. Preconditions: green, trustworthy test suite (invest here first if coverage is thin — request specs over critical flows give the most upgrade confidence per hour), deprecation warnings at zero on the current version (treat deprecations as failing CI; they are the next version's errors), Ruby version compatible with the target Rails.
  2. One minor version at a time. Never leapfrog. 6.1 → 7.0 → 7.1 → 7.2 → 8.0. Each hop: read the upgrade guide + CHANGELOG, run rails app:update and diff every config choice consciously.
  3. Dual boot with the next_rails gem (Shopify's approach): a Gemfile.next lets CI run the suite against current and next Rails on every PR. The team fixes incompatibilities incrementally on main — no long-lived, ever-rotting upgrade branch, no big-bang merge. This is the single most important technique to name.
  4. new_framework_defaults_X_Y.rb: upgrade with old defaults first, ship, then flip defaults one at a time — each flag is its own small, revertable change (e.g., cache_format_version, cookie serializer changes need rolling-deploy-safe handling).
  5. Known landmines per era: 6.x→7.0: Zeitwerk becomes mandatory (fix autoloading violations with zeitwerk:check beforehand), Spring/listen changes, button_to/UJS → Turbo migration decisions; 7.x: Sprockets → importmap/esbuild/Propshaft asset story, ActiveRecord::Base.connection deprecations; 8.0: Sprockets fully out (Propshaft default), the Solid trifecta (Queue/Cache/Cable) as optional infra simplifications, built-in authentication generator.
  6. Rollout: canary deploy the upgraded version to a slice of traffic, watch error rates and latency, keep the revert path one deploy away.

Dependency and CVE management as an ongoing system

  • Automate detection: bundler-audit in CI (fails builds on known CVEs), Dependabot/Renovate for continuous small bumps — 40 one-line bump PRs over a year beat one terrifying "update everything" PR. GitHub security advisories + the ruby-advisory-db.
  • Patch-severity triage: a critical Rails/Rack CVE is a drop-everything same-day patch (Rails maintains security releases for the supported series — being on an EOL Rails means no patches exist, which is the strongest business argument for staying current). Frame it that way to non-technical stakeholders: staying upgraded is security posture, not developer vanity.
  • Deprecated/abandoned gems: quarterly dependency review; for each risky gem, decide — replace, vendor + own it, or PR upstream. Check pulse before adopting anything new (maintainers, release cadence, issue responsiveness). Wrap third-party APIs behind your own adapter class so a gem swap touches one file.
  • Ratchet, don't boil the ocean: RuboCop's --auto-gen-config TODO file to freeze current violations and forbid new ones; same idea for deprecation counts and coverage — metrics may only improve.

Topic 4 — Engineering Leadership & Team Delivery

Prepare every answer here as a STAR story (Situation → Task → Action → Result) with a real example, real numbers where possible, and a closing "what I learned / what I'd do differently." Below: the position to take, plus story scaffolds to fill with your own history before the interview.

4.1 Handling Technical Debt

The position

  • Translate debt into product language. PMs don't fund "refactoring"; they fund outcomes. "The checkout module's defect rate means every feature there takes 3× estimate and causes ~2 incidents/quarter; six weeks of restructuring makes the Q3 roadmap in that area 40% cheaper" is a negotiable proposition. "The code is ugly" is not.
  • Make debt visible and priced: a lightweight debt registry (tagged tickets, one-line cost/impact each), reviewed at planning — debt items compete for capacity with features using the same currency (risk, velocity, incidents).
  • Prefer continuous payment over big rewrites: a standing capacity allocation (~15–20% of each cycle) plus the Boy-Scout rule for opportunistic cleanup, and attach refactors to feature work in the same area ("while we're building discounts in checkout, we pay down these two items — it de-risks the feature"). Big-bang rewrites are almost always the wrong answer; say so, and name strangler-fig incremental replacement as the alternative.
  • Not all debt is worth paying. Debt in code you're about to delete or that changes once a year can sit. Prioritize by interest rate: how often the team touches it × how much it hurts each time.
  • Some debt is a good trade — taken knowingly, to hit a market window, with the payback scheduled. The failure mode is unknowing or unrecorded debt.
Story to prepare: a time you negotiated refactoring time with a PM — include the metric that made the case (cycle time, incident count, estimate misses), how you scoped it into shippable slices, and the measured result.

4.2 Mentorship & Growth

The position

  • Grow people by delegating real ownership, sized one notch beyond comfort, with safety nets (design review before code, checkpoints, you as escalation) — not by handing them the easy tickets. Measure yourself as a lead by what the team ships when you're on vacation.
  • Code review as the primary teaching channel: explain the why behind every suggestion, distinguish blocking from optional comments (nit: prefixes), review with questions not verdicts, and move recurring debates into automation (RuboCop) or the team conventions doc so humans stop re-litigating style. On RuboCop adoption in a legacy app: adopt a community baseline (standard or rubocop-rails-omakase), --auto-gen-config the existing violations into the TODO file so day one is green, autocorrect the safe cops in one mechanical PR, ratchet the rest. The goal is ending style debates, not maximal strictness — a rule the team resents gets disabled or ignored.
  • Deliberate skill-building: pairing rotations (especially on incidents and gnarly bugs — debugging is the least-taught skill), lightweight design docs for anything over a few days (writing forces clarity, review spreads context), internal tech talks, and giving mid-level engineers the lead on scoped projects with you as consultant.
  • Blameless post-mortems that actually work: timeline, contributing factors (plural — never a single "root cause: human error"; if a human could break it, the system lacked a guardrail), action items with owners and dates, and — the part most teams skip — reviewing whether past action items were done. A post-mortem whose actions evaporate teaches the team the ritual is theater.
Stories to prepare: (1) a specific junior/mid engineer you grew — what gap you saw, what you delegated, the checkpoint structure, where they are now; (2) a post-mortem you ran that changed something durable (a guardrail, a process), not just "we'll be more careful."

4.3 Conflict Resolution

The scenario: two senior engineers fiercely disagree on architecture

The answer that lands is a process, not a personality claim:

  1. Separate the people from the decision, early. If it's turned heated in public channels, talk to each privately first — often the technical dispute is carrying something else (ownership anxiety, past decisions being relitigated, one person feeling unheard).
  2. Force shared criteria before comparing solutions. Get both to agree on what the decision is optimizing for — latency? operational simplicity? time-to-market? team skill fit? Most "fierce" disagreements are two people optimizing different unstated objectives; making the objectives explicit resolves half of them outright.
  3. Make it concrete and cheap: each writes a one-page proposal against those criteria (writing dissolves rhetorical advantage), and if genuinely close, timebox a spike — a week of prototype data beats a month of meetings.
  4. Decide, record, commit. Consensus is the goal but not a requirement; stalemates are the lead's job to break. Record it as an ADR (Architecture Decision Record) — context, options, decision, consequences — so it isn't re-argued in six months and the "losing" author's option is documented as seriously considered. Then explicitly invoke disagree and commit, and give the engineer whose proposal lost a meaningful role in the chosen path (they're the best-placed critic to make it robust).
  5. Watch the aftermath. Quiet disengagement from the "loser" is the real cost of badly resolved conflict; a 1:1 two weeks later ("you pushed hard for X — how's Y feeling in practice?") signals the dissent was valued. And if reality proves the decision wrong, reverse it publicly and cheerfully — that's what makes people willing to commit to decisions they disagreed with next time.
General conflict framing to reuse: healthy teams argue about ideas; my job is keeping arguments (a) evidence-based, (b) bounded in time, and (c) ending in a recorded decision someone owns. The anti-patterns I actively prevent: decisions by attrition (loudest/most persistent wins), decisions by silence (nobody decides, architecture accretes), and permanent relitigation.

Rapid-fire behavioral prompts to have stories for

  • A technical decision you made that turned out wrong — and how you handled discovering it. (Owning it quickly and publicly is the entire point of the question.)
  • A time you pushed back on a deadline — and a time you accepted one you disliked and made it work.
  • An underperforming teammate: how you diagnosed (skill vs. clarity vs. motivation vs. life circumstances) before acting, what support you set up, and the honest outcome either way.
  • How you balance your own IC coding time against unblocking the team (the classic tech-lead tension — good answer: your code goes on the non-critical path; the team's throughput is your output metric).
  • Disagreeing with your own manager or a strong PM — showing you can advocate upward with data and still commit.
